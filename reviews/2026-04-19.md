### `slack_app.py` — Routing, slash commands, file handling, feedback loop

Full `_route_request()` function (keyword-based routing to `sales`/`marketing`/base). 10 slash commands registered. **Yesterday's follow-ups — status:**
1. ✅ `bot_user_id` caching — `_get_bot_user_id()` function with `_BOT_USER_ID` global. One API call, then cached.
2. ✅ Bot-id guard — `handle_reaction` now checks `if msg_user != bot_user_id: return` correctly.
3. ✅ Skill threading through DM feedback path — `skill=state.get("skill")` in `handle_dm` correctly passes skill to `append_feedback`. Skill stored in pending state via `handle_reaction`.

**New risk flags:**
- **Concurrency race on same thread** (`slack_app.py:203-235`): `_run_and_reply` loads history with the lock, then releases it before the API call. If two Slack messages arrive on the same thread before either finishes, both load the same history snapshot, both append, both run_agent, and the second `mem_save` overwrites the first. The lock only guards the load step, not the full read-modify-write cycle. Acceptable at low volume; becomes a real bug under concurrent thread usage. Fix: hold a per-`thread_ts` lock or implement optimistic write-conflict detection.
- **`_send_chunked` doesn't split oversized single lines** (`slack_app.py:242-264`): if a single line exceeds 2900 chars (e.g., a JSON snippet, long URL list), it posts the entire line as one chunk and Slack rejects messages over 3001 chars. Edge case but real with long tool-result dumps.
- **`_slash_handler` thread_ts fallback** (`slack_app.py:293`): `body.get("thread_ts", body.get("ts", channel))` — slash command payloads don't carry `thread_ts`. Replies go to the channel root, not an existing thread. This was pre-existing but worth documenting as known behavior.
- **`map_objections` has no slash command** — the skill exists in `prompts.py` and routing, but users can only reach it via auto-routing keywords (`bezwaar`, `objection`, `weerstand`). Consider `/objections` for discoverability.
