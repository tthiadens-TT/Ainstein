## Code changes

### `agent.py`
Multi-agent support added: `agent_type` parameter accepted, `AGENT_SYSTEM_PROMPTS` dict lookup, prompt caching for both system blocks via `cache_control`. Max-iterations guard with nudge message and forced final call. Per-iteration timing and character-count logging added.

**Issues:**
- `import time as _time` and `from prompts import AGENT_SYSTEM_PROMPTS` both live inside the `while True:` loop (lines 110 and 68). Python caches imports so there's no performance impact, but it reads like re-execution on every iteration and will trigger linting warnings. Move both to module top.
- The max-iterations nudge is appended to the caller-owned `messages` list as a side effect — callers (like `chat_loop`) don't expect this mutation and will include the injected user turn in conversation history on any subsequent call. Low-probability issue for the current single-turn Slack usage, but fragile for multi-turn CLI use.
- Max-iterations fallback response is in Dutch; the `print` log on the same path is in English. Keep one language throughout for consistent log monitoring.

### `feedback.py`
New module. Implements the full 👎 → feedback-capture flow: in-memory pending state keyed by `(channel, user_id)`, `register_pending` / `has_pending` / `pop_pending` guards, and `append_feedback` that creates `07_Feedback/gaps.md` and appends structured entries.

**Issues:**
- **Race condition in `append_feedback`:** The `fresh = not gaps_file.exists() or gaps_file.stat().st_size == 0` check (line 93) runs *outside* the `_lock`. Two simultaneous feedback events can both compute `fresh = True` and both write `_HEADER`, producing a duplicate header at the top of `gaps.md`. Fix: move the `fresh` check inside the `with _lock:` block.
- In-memory pending state resets on bot restart — documented as an acceptable loss. Fine.

### `routing.py`
New module. Deterministic `route_request(text) → (agent_type, skill)` extracted from `slack_app.py`. Checks debrief signals first, then marketing, then sales, then base fallback.

**Issues:**
- `"risico"` in the `map_objections` keyword list can false-positive on messages that discuss risk in a non-objection context (e.g., `"wat is het risico van dit voorstel"` routes to `map_objections` rather than `build_proposal`, since `"voorstel"` is in `sales_words` and `"risico"` hits the objections sub-check first). Borderline — may be acceptable, but worth monitoring.
- The test comment in `test_bezwaar_routes_to_map_objections` claims `"bezwaren"` has a "single-a" — incorrect. `"bezwaar" in "bezwaren"` is `True` (substring match), so `"bezwaren"` also triggers `map_objections`. This is likely intentional but the comment is misleading.

Large expansion: `SALES_SYSTEM_PROMPT`, `MARKETING_SYSTEM_PROMPT`, `AGENT_SYSTEM_PROMPTS` dict, and 8 new skill prompts (`qualify_lead`, `prepare_discovery`, `map_objections`, `sharpen_positioning`, `create_content`, `adapt_messaging`, `client_discovery_debrief`, `debrief_to_messaging`). All prompts are well-structured and consistent with the Minkowski operating rules.

**No logic errors found.** Content quality is high.

### `slack_app.py`
Major update. New slash commands for both agents (`/qualify`, `/discovery`, `/objections`, `/debrief`, `/content`, `/positioning`, `/debrief-messaging`). DM file attachment handling with Slack-host validation for token security. `handle_reaction` captures 👎 and triggers feedback loop. Thread-scoped locking via `_thread_locks`. Chunked message sending for long responses.

**Issues:**
- The Slack-host validation (`host in _SLACK_FILE_HOSTS`) is correct — `urlparse().hostname` returns only the hostname component, so subdomain spoofing is not possible. No issue.
- `_slash_handler` sets `thread_ts = body.get("thread_ts", body.get("ts", channel))`. Slash commands do not carry a real `thread_ts`, so this falls back to `channel` as the key — which effectively groups all slash responses per-channel rather than per-thread. For the current single-session use this is fine, but could cause unexpected memory sharing if two people use a slash command in the same channel simultaneously.
- `_BOT_PLACEHOLDER_PREFIXES` uses `str.startswith()` correctly; no issue.
