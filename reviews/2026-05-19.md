## For the developer [dev]

### NIEUW — `deploy.yml:21` hardcodes `git pull origin dev` voor beide branches

Het CI/CD-workflow trigt op zowel `dev` als `main`, maar het deploy-script doet altijd:

```yaml
git pull origin dev
```

Dit betekent dat een push naar `main` ook `dev` deployt, niet `main`. In de huidige situatie zijn `dev` en `main` dicht bij elkaar, dus dit valt waarschijnlijk niet op — maar zodra `main` achterloopt op `dev` (of andersom) deployt het de verkeerde code. Correcte fix:

```yaml
git pull origin ${{ github.ref_name }}
```

Of — als je altijd `dev` wil deployen — beperkt de trigger dan tot alleen `dev`. Referentie: `.github/workflows/deploy.yml:21`.

Bijkomend: er is geen health check na `systemctl restart ainstein`. Als de service failт te starten, rapporteert de workflow toch "Deploy done". Overweeg:
```bash
sudo systemctl is-active ainstein || (systemctl status ainstein; exit 1)
```

### Two-layer caching — `tools.py:48–74` (positief, één opmerking)

De SQLite-caching is correct geïmplementeerd: WAL-mode, `INSERT OR REPLACE`, stale-entry cleanup, `check_same_thread=False`. `*.db` staat al in `.gitignore` — `file_cache.db` wordt niet gecommit. ✓

Één aandachtspunt: `_CACHE_DB` is een module-level global die zonder lock wordt geïnitialiseerd. Onder hoge concurrency (meerdere threads die tegelijk `_cache_db()` aanroepen vóór `_CACHE_DB` is gezet) kan dubbele init optreden. In de huidige enkele-thread Slack-app is dit geen probleem, maar als async/threading uitgebreid wordt: voeg een `threading.Lock` toe rond de init.
