`_check_drive_write_access()` is a good pattern — surfaces permission failures at startup instead of at first user action.

4. **`tools.py:_check_drive_write_access` — orphan file risk on delete failure**
   If the create succeeds but the delete API call fails (network hiccup, transient 5xx), a `_ainstein_write_test` file is left in Drive root. The function has no retry or cleanup for this case. Over time and across deploys, these could accumulate.

### Carry-forward: slack_app.py / error handling

5. **`slack_app.py:241` — raw exception passed into fallback prompt context**
   `f"Er was een technisch probleem bij het beantwoorden: {e}\n\n"` includes the raw Python exception in the context sent to the fallback `run_agent`. The new brain.md rule says never show raw technical errors to users.
